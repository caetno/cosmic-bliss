When the user has stated *what* they want and only the *how* is open, just do it. Don't enumerate options and ask them to pick. Reserve discussion for genuinely ambiguous architectural calls.

**Why:** During the BoneMap session, after the user said "make it available via Quick Load at least," I responded with two implementation options and asked which one to do. They replied "I told you what I want and what I need, please do it." Same pattern repeated when I asked whether to bake the BoneMap entries — they had to push past another confirmation step.

**How to apply:** When the user names a goal and the path is unambiguous (or has one obvious-best choice), execute. If genuinely torn between approaches with material trade-offs, briefly state the trade-off in the same response that acts on the most defensible default — don't block on confirmation. Save preambles like "Want me to…?" for changes with real reversibility cost or scope ambiguity.
