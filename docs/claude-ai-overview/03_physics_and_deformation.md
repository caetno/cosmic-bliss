# Physics and Deformation

The realism focus area. PBD-based, CPU-bound, on purpose.

## Why CPU PBD

The hot path runs on the CPU because:
- The math isn't the bottleneck at the data scales we're at (≤ ~320 PBD particles total — chain + rim + centerline combined per character)
- Going GPU would tax CPU↔GPU sync for behaviour drivers, gizmos, Marionette reciprocals, the future stimulus bus
- A GTX 970 doesn't make GPU compute worth it for this scale

Decided not to go GPU for the foreseeable future. See `07_engine_constraints.md` for the broader hardware-target rationale.

## PBD foundation

XPBD (Extended Position-Based Dynamics; Macklin 2016) with compliance + per-constraint Lagrange-multiplier accumulators. Pattern:

```
predict (apply gravity, damping, external forces)
for each substep:
  for each iteration:
    project constraints (distance, bending, contact, friction, ...)
finalize (compute implicit velocity from position delta)
```

Per-constraint compliance maps from a public `stiffness ∈ [0, 1]` knob via a log-spaced curve (`stiffness=1 → compliance=1e-9` near-rigid; `stiffness=0 → compliance=1e-3` very soft). Lambdas accumulate **across iterations within a substep** (XPBD invariant) and **reset per substep** (Obi convention). Cross-tick warm-starting of contact lambdas is currently absent — a known instability seed under low friction; see `08_current_state.md`.

The **Obi 7.x Unity asset source** (locally available, not redistributable) is the implementation reference for constraint forms, lambda-accumulator patterns, and the Jacobi-with-atomic-deltas-and-SOR pattern. The user has the source locally; canonical references in the architecture doc cite specific compute files.

## The tentacle

A 16-particle PBD chain anchored at the base. Constraints:
- **Distance** (XPBD compliance form)
- **Bending** (chord between i and i+2; full 3-particle angle form is Phase 9 polish)
- **Target pull** — singleton tip target + distributed pose targets, both subject to `target_softness_when_blocked` so the chain *gives* to obstacles instead of fighting them
- **Anchor** — hard pin, optionally rigidifying the first N particles to a transform

Per-particle state: position, prev_position, inv_mass, **girth_scale** (volume-preserving radial scaling), **asymmetry** (Vector2 ellipse shape).

The mesh is generated by `TentacleMesh : PrimitiveMesh` from a base shape (linear/curve taper, twist) plus a stack of feature modifiers — Knots, Ribs, WartClusters, SuckerRows, Spines, Ribbons, Fins. Each feature contributes geometry and (since slice 5H) silhouette data.

## Feature silhouette (2026-05-05)

Features bake a **2D R32F texture** (256 axial × 16 angular) of outward radial perturbation in metres. Sampled at contact time and **added to the smooth `girth_scale × collision_radius`** to derive the effective collision threshold. This is what makes warts displace the orifice rim, ribs catch on edges, sucker pits reduce contact pressure, etc.

Negative values are legal (sucker pits, scars). Type-1 / type-2 / type-4 contacts all consume the silhouette; type-3 (canal walls, in design) wires the same sampler.

The body-frame θ axis is rotation-minimizing-frame parallel-transported from the anchor — stable under bending but doesn't track twist. Twist tracking is a known follow-up.

## Orifices — rim particle loop model

Replaced an earlier 8-direction discrete-ring model. Now each orifice is a **closed loop of N PBD particles** in a host-bone-relative Center frame, with:

- XPBD distance constraints around the loop
- XPBD volume constraint on enclosed area (planar-shoelace reduction of Obi's 3D triangle-fan-from-origin volume form, since the rim is intentionally planar)
- Per-particle XPBD spring-back to authored rest position
- **4P-A — anisotropic distance.** Compression branch is near-rigid; stretch branch is soft (~1000× lower compliance). Anatomical flesh is incompressible but stretch-compliant.
- **4P-B — strain-stiffening J-curve.** `strain = displacement / characteristic_length`; effective compliance scales as `1 / (1 + α s² + β s⁴)`. Collagen-style nonlinear stiffening.
- **4P-C — plastic memory.** Runtime `plastic_offset` accumulates toward sustained displacement and decays back toward zero, clamped. Defaults are memory-neutral; bumping accumulate > recover gives long-term remodeling.

**Multi-loop per orifice** is supported (outer + inner rings; jewelry; multi-sphincter). Loops are mechanically independent; inter-loop coupling springs are a deferred polish item.

**Host-bone soft attachment** — the orifice is parented to a Marionette skeleton bone via a cached transform. The Center frame is the bone's world transform × `host_bone_offset`. Rim positions are stored in Center-frame and projected through this transform per tick.

**EntryInteraction** — per-(tentacle, orifice) state object with a lifecycle (created on first crossing, retired after grace period, hysterically persistent through brief disengagements). Tracks geometric facts (entry point, depth, axial velocity, approach angle, particles-in-tunnel) plus persistent slots (grip_engagement, in_stick_phase, ejection_velocity, per-rim-particle damage / pressure / friction accumulators).

**§6.3 reaction-on-host-bone** — the third law closes here. Per rim particle in contact: radial force + axial-wedge force + tangential friction force are summed and routed via `PhysicsServer3D::body_apply_impulse` to the host bone. This is what physically realizes "the tentacle is suspending the orifice's host body" instead of leaving the rim's reaction in the air.

The wedge math uses the normalized form `−p × drds_outward / sqrt(1 + drds_outward²)` (= `−p × sin θ`), bounded by `p` at near-vertical flanges. The unnormalized linearization blows up exactly at the geometry that matters most.

## Canals (5E/5F/5G — in design)

For uterus / anus / vagina / future stomach. The interior shape changes too radically tick-to-tick to model as a static mesh; the rim model doesn't extend gracefully to long curved cavities.

The canal interior model uses:
- A **2D `tunnel_state` RGBA32F texture** (32 axial cells × 8 angular cells) carrying per-cell rest_radius / dilation / pressure / damage
- A **PBD centerline particle chain** (M = 12 particles) tracking the canal's curve under tentacle deflection
- Per-vert bake of `(s, θ, rest_radius, normal)` baked into vertex attributes, **skipping standard skin weights**
- Bilateral wall/centerline split for type-3 contacts: tentacle particle ↔ canal wall (radial perturbation from tunnel_state) AND tentacle ↔ centerline (bending the canal)

Curved canals are supported by the centerline chain — author the canal as a curved bone chain, the chain's bending stiffness controls how much it resists deflection. Sacs (uterus, stomach) are canals with `closed_terminal = true`.

This work is the next gate for TentacleTech (5E infrastructure → 5F dynamics → 5G muscle activation).

## Tentacle ↔ ragdoll coupling

Each tentacle PBD tick:
1. Probe environment with sphere-shape queries → contacts (point, normal, RID, depth) per particle, up to `MAX_CONTACTS_PER_PARTICLE`
2. Iterate constraints (distance, bending, contact, friction)
3. After tick, walk friction_applied per contact slot and apply reciprocal impulses to colliding bodies via `body_apply_impulse`

When the colliding body is a `PhysicalBone3D` (Kasumi's leg, hip, pelvic floor), the bone's joint network responds — angular springs deflect, then restore. The tentacle's next tick re-probes against the moved bone.

This **PBD↔ragdoll feedback loop** is currently the active design problem. Under low friction (high lubricity) the loop becomes a coupled oscillator: contact normal flips per tick, reciprocal impulse direction flips, bone wobbles. Investigation status in `08_current_state.md`.

## Realism levers we have

- Per-particle `girth_scale` (volume preservation) and `asymmetry` (ellipse)
- Feature silhouette (per-(s, θ) outward perturbation)
- Anisotropic rim distance (compression vs. stretch)
- J-curve strain-stiffening (collagen-like)
- Plastic offset (orifice memory)
- Substep / iter count (convergence quality)
- Per-mood `tentacle_lubricity`, `pose_softness_when_blocked`, base / contact stiffness
- Friction modulation (μ_s × (1 − lubricity), kinetic ratio)
- `body_rhythm_phase` coupling for rhythmic motion
- `RhythmSyncedProbe` modifier on tentacles for body-driven thrust

## Realism levers off-limits or underused

- **Off-limits — fragment-shader-heavy effects.** GTX 970 doesn't have the budget. No screen-space subsurface, no volumetric anything, no parallax-occluded suckers per pixel, no full-screen post-stack.
- **Off-limits — GPU compute** for PBD itself. CPU sync tax with behaviour drivers + Marionette + bus would eat the win. (Future GPU compute for fluids/slime via Tenticles is fine — that subsystem doesn't share state per tick.)
- **Underused — substepping.** Default `substep_count = 1`. Obi convention is 4 substeps × 1 iter; we currently ship 1 × 4. Flipping is a candidate fix for the wedge stability problem.
- **Underused — lambda warm-starting.** Currently lambdas reset per substep / per tick. RID-keyed warm-start across ticks for stable contact pairs is on the table.
- **Underused — contact-point persistence in body-local space.** Currently re-probe every tick. Anchoring contact in body-local frame for stable RIDs would smooth out moving-collider cases.
- **Underused — multi-tentacle multi-orifice coupling.** Spec defines cap-3 per orifice; not enforced; storage chains and oviposition / birthing are deferred behaviours that the architecture supports.

## Constraint patterns we deliberately avoid

- **No `SoftBody3D`.** Godot's built-in is unsuitable for what we need.
- **No SSBOs in spatial shaders.** Godot 4.6 doesn't support them. Use RGBA32F data textures instead.
- **No boolean rejects.** A behaviour that "shouldn't happen" must be expressed via stiffness / friction / grip / damage, not a hard `if`-statement. (Cross-cutting principle, not just a code rule — applies to design discussions too.)
- **No per-frame `ArrayMesh` rebuilds.** Vertex shader deformation only.
- **No per-frame `ShaderMaterial` allocation.** One per tentacle instance, mutate uniforms.
- **No `MeshDataTool` in hot paths.** Ever.
